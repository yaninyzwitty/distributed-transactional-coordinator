# Plan: Justfile Recipe System for Distributed Transactional Coordinator

## Overview

Create a `Justfile` at the project root with top-level variables and two modular recipe files under `just/`:
- `just/gen.just` — code generation (sqlc, protobuf via buf, mocks)
- `just/db.just` — database migrations, seeding, and related tasks

All tools are managed via `mise.toml` and secrets via Doppler.

---

## 1. Root `Justfile`

**File**: `Justfile` (project root)

### Variables (top-level)
```
DATABASE_URL := `doppler secrets get DATABASE_URL --no-file`
```

### Recipes
- `default` — show help (list available recipes)
- Forward to `gen.just` and `db.just` recipes using Just's delegation syntax (`@just --justfile ...`)

### Design decisions
- The root `Justfile` is the single entry point for `just` commands
- `DATABASE_URL` is fetched live from Doppler each time a recipe needs it (not cached), ensuring secrets are always current
- `@mise.toml` is used as the tool version source — all tools (`go`, `sqlc`, `buf`, `doppler`, `goose`, `mockgen`) are pinned there
- No shell aliases or PATH overrides — mise handles tool shimming

## 2. `just/gen.just`

**File**: `just/gen.just`

Purpose: Code generation recipes for sqlc, protobuf (buf), and mocks.

### Recipes

| Recipe | Description |
|--------|-------------|
| `default` | Show help for available gen recipes |
| `sqlc` | Run `sqlc generate` using `sqlc.yaml` config (generates `internal/repository/` from `db/queries/` + `db/migrations/`) |
| `proto` | Run `buf generate` using `buf.gen.yaml` (generates Go protobuf + gRPC stubs into `gen/`) |
| `buf-lint` | Run `buf lint` against `proto/` directory |
| `buf-breaking` | Run `buf breaking` against `proto/` directory |
| `mocks` | Run `mockgen` for interfaces in `internal/repository/` (generated by sqlc), output to `mocks/` |
| `all` | Run `sqlc`, `proto`, `mocks` in sequence |

### Tool confirmations (from `mise.toml`):
- `sqlc` → `1.31.0` (mise key: `sqlc`)
- `buf` → `1.69.0` (mise key: `buf`)
- `mockgen` → `0.6.0` (mise key: `go:go.uber.org/mock/mockgen`)
- `go` → `1.26.2` (available for `go generate` if needed)

### Notes
- `proto/` directory does not yet exist — `buf` recipes will fail until proto definitions are added (handled separately)
- `sqlc` reads config from project-root `sqlc.yaml`
- `mockgen` targets `internal/repository/` package (sqlc-generated interfaces), output goes to project-root `mocks/` directory

## 3. `just/db.just`

**File**: `just/db.just`

Purpose: Database operations — migrations, seeding, status checks.

### Recipes

| Recipe | Description |
|--------|-------------|
| `default` | Show help for available db recipes |
| `migrate` | Run `goose up` applying all pending migrations in `db/migrations/` |
| `migrate-down` | Run `goose down` to roll back the last migration |
| `migrate-status` | Run `goose status` to show applied/pending migrations |
| `migrate-reset` | Run `goose reset` to re-run all migrations from scratch |
| `seed` | Run `goose seed` from SQL seed files in `scripts/` directory |
| `status` | Show migration status + verify database connectivity |

### Tool confirmations (from `mise.toml`):
- `goose` → `v3.27.1` (mise key: `go:github.com/pressly/goose/v3/cmd/goose`)
- `doppler` → `3.76.0` (mise key: `doppler`) — used to fetch `DATABASE_URL`

### Notes
- Every recipe that touches the database fetches `DATABASE_URL` from Doppler at invocation time
- `goose` is configured to use the `db/migrations/` directory for SQL migrations
- Seed files live in `scripts/` directory as `.sql` files, run via `goose seed`

## 6. Just Best Practices Applied

Based on justfile best practices:
- `.justfile` or `Justfile` at project root (using `Justfile` for case-insensitive OS compatibility)
- Top-level variables for configuration (`DATABASE_URL`)
- Modular recipe files under `just/` directory for separation of concerns
- `@` prefix for private/internal recipes where appropriate
- `default` recipe in each file that shows help
- Comments on every recipe for `just --list` readability
- Use `#!/usr/bin/env just --justfile` shebang is NOT used (just runs from project root)
- Each recipe is self-documenting with description comments
